# DrawOp Parameter Diff: Widget-Level Divergence Diagnosis

## Objective

Build a mechanical diagnostic tool that compares the exact paint-call parameters generated by Rust widget rendering against C++ widget rendering, op-by-op, for any golden test. This produces ground truth about WHERE widget-level parameter divergence occurs, without forming hypotheses from code reading.

## Background

208/241 golden tests pass. 33 fail. Every individual paint primitive (PaintBorderImage, PaintRoundRect, PaintPolyline, PaintText, blend ops, area sampling, gradients) is proven byte-identical via FFI harness testing. The remaining divergence is at the widget level: Rust widgets compute different parameters (coordinates, colors, clip rects, canvas colors) than C++ widgets, and pass those different parameters to the proven-correct paint functions.

Prior sessions spent significant effort forming hypotheses from code reading — every hypothesis was wrong. The FFI harness approach (generate inputs, call both implementations, compare outputs) is the only method that has produced ground truth. This spec extends that approach to the widget parameter level.

### What Has Already Been Fixed (Do Not Re-Investigate)

- Area sampling carry/pCy (b272bf8)
- 9-slice transform unification (4183064)
- EXTEND_EDGE for PaintBorderImageColored (d667fc7)
- PaintRect/paint_image_full/PaintImageColored iy2=truncate (c8383a3, 7edc3a7, d223d8e)

## Architecture

```
Rust golden test ─→ DrawOp recording ─→ serialize ─→ {name}.rust_ops.jsonl
C++ golden gen   ─→ LoggingPainter   ─→ serialize ─→ {name}.cpp_ops.jsonl
                                          │
                                    diff_draw_ops.py
                                          │
                                  per-op divergence report
```

## Component 1: Rust DrawOp JSONL Serialization

### What

Add a function `dump_draw_ops(name: &str, ops: &[DrawOp])` to the golden test common module. It serializes each DrawOp as a single JSONL line to `target/golden-divergence/{name}.rust_ops.jsonl`.

### Trigger

Environment variable `DUMP_DRAW_OPS=1`. When set, golden tests that use `SoftwareCompositor::render()` or direct painter rendering additionally record DrawOps and dump them.

### JSONL Format

Each line is a JSON object:

```json
{"seq":0,"op":"SetCanvasColor","color":"00000000"}
{"seq":1,"op":"PaintRect","x":0.0,"y":0.0,"w":400.0,"h":14.814814814814815,"color":"4060a0ff","canvas_color":"00000000"}
{"seq":2,"op":"PaintBorderImage","x":10.0,"y":5.0,"w":380.0,"h":280.0,"l":20.0,"t":20.0,"r":20.0,"b":20.0,"src_l":286,"src_t":286,"src_r":286,"src_b":286,"alpha":255,"canvas_color":"00000000","which_sub_rects":495}
```

Rules:
- `seq`: monotonically increasing integer, starting at 0
- `op`: exact DrawOp variant name
- Colors: lowercase 8-hex-digit RGBA (e.g., `"4060a0ff"`)
- Floats: full f64 precision (no rounding)
- Image pointers: omitted (not comparable cross-language). Instead include image dimensions as `img_w`, `img_h`, `img_channels`.
- Vertices (polygon/polyline): inline array of `[x, y]` pairs

### Implementation Location

- Serialization: `crates/eaglemode/tests/golden/common.rs` — new function `dump_draw_ops`
- Recording: Modify `render_testpanel` and similar render helpers to optionally record DrawOps when env var is set. For compositor-based tests, the compositor's paint phase already walks the panel tree — add a DrawList recording pass before or alongside the pixel rendering.

### Recording Approach for Compositor Tests

The `SoftwareCompositor::render()` currently walks the panel tree and paints directly to tiles. To record DrawOps:

1. Create a painter with `PaintTarget::DrawList(&mut ops)` and the same clip/transform as the main render.
2. Run the same panel tree paint walk into this recording painter.
3. Dump the recorded ops.
4. The normal pixel rendering proceeds unchanged (tests still pass/fail as before).

This is a separate, additional paint pass — it does not modify the rendering path.

## Component 2: C++ LoggingPainter

### What

A C++ class `LoggingPainter` that wraps an `emPainter` and logs every paint call's parameters to JSONL before forwarding to the real painter.

### Location

`crates/eaglemode/tests/golden/gen/logging_painter.h` — header-only.

### Interface

```cpp
class LoggingPainter {
public:
    LoggingPainter(emPainter& real, FILE* log);

    // Intercepts — log params, then forward:
    void PaintRect(double x, double y, double w, double h, emColor color, emColor canvasColor=0);
    void PaintRoundRect(double x, double y, double w, double h, double r, emColor color, emColor canvasColor=0);
    void PaintBorderImage(double x, double y, double w, double h,
        double l, double t, double r, double b,
        const emImage& img, int srcL, int srcT, int srcR, int srcB,
        int alpha=255, emColor canvasColor=0, int whichSubRects=0757);
    void PaintText(double x, double y, const char* text, double charHeight,
        double widthScale, emColor color, emColor canvasColor=0);
    void SetCanvasColor(emColor c);
    // ... all other paint methods that golden tests exercise

private:
    emPainter& real_;
    FILE* log_;
    int seq_;
};
```

Each method writes one JSONL line with the same format as the Rust serialization, then calls the corresponding method on `real_`.

### Integration

In `gen_golden.cpp`, for tests that use `render_and_dump_sized()`:

1. The `DoPaintView(p, 0)` call goes through the C++ view/panel paint tree using a real `emPainter`.
2. To capture ops: wrap the painter in a `LoggingPainter` before passing to `DoPaintView`.
3. The logging painter outputs `{name}.cpp_ops.jsonl`.

For tests that use raw painter calls (like `gen_cosmos_item_border`): wrap the painter similarly.

### Scope

Only the paint methods that appear in failing golden tests need logging. Based on the 33 failures, the required methods are:

- `PaintRect`
- `PaintRoundRect`
- `PaintBorderImage`
- `PaintBorderImageColored`
- `PaintImageFull` / `PaintImage`
- `PaintImageColored`
- `PaintText`
- `PaintTextBoxed` (if used)
- `PaintEllipse`
- `PaintPolygon`
- `PaintSolidPolyline`
- `SetCanvasColor`

State ops (`PushState`, `PopState`, `ClipRect`, `SetOffset`) are important for context but harder to intercept from C++. The C++ `emPainter` constructor chain sets up transforms — these would need to be logged as synthetic ops when the painter is created with a sub-region.

### C++ Painter Interception Challenge

`emPainter` in C++ is not virtual — its paint methods are non-virtual member functions. The logging wrapper cannot inherit and override. Two options:

**Option A (Recommended): Preprocessor interception.** Define macros that rename `emPainter` methods and insert logging. This is ugly but requires zero changes to the C++ library.

**Option B: Source-level printf.** Add `fprintf` calls directly to the C++ `emPainter` methods in a local copy. The golden generator already links against a local build of Eagle Mode — modify the local copy's `emPainter.cpp` to log when an env var is set.

**Chosen: Option B.** Modify `~/git/eaglemode-0.96.4/src/emCore/emPainter.cpp` to add conditional logging (guarded by a global `FILE* g_draw_op_log`). This is cleaner than macro hacks and provides exact parameter values at the point of use. The modifications are:

1. A global `FILE* g_draw_op_log = nullptr;` and `int g_draw_op_seq = 0;`
2. At the top of each paint method, if `g_draw_op_log != nullptr`, write a JSONL line
3. In `gen_golden.cpp`, open the log file before rendering, close after

This touches C++ source but in a minimal, guarded way that doesn't affect normal operation. The modifications are diagnostic-only and should be reverted (or left guarded) after diagnosis is complete. They do not affect normal Eagle Mode builds since the global log pointer defaults to `nullptr`.

## Component 3: Diff Script

### What

`scripts/diff_draw_ops.py` — reads two JSONL files (C++ and Rust), aligns by sequence number, and reports parameter-level differences.

### Algorithm

1. Load both files into lists of dicts
2. For each index `i`, compare `cpp_ops[i]` vs `rust_ops[i]`:
   - If op types differ: report "OP MISMATCH at seq {i}: C++ {type} vs Rust {type}" and stop
   - If op types match: compare each parameter
   - For floats: report if `abs(cpp - rust) > 1e-10`
   - For ints/colors: report if not equal
3. If one file has more ops than the other: report "OP COUNT MISMATCH"
4. Output: table of divergences sorted by sequence number

### Output Format

```
=== cosmos_item_border: 3 divergences in 7 ops ===
seq  op           param          C++              Rust             delta
---  -----------  -------------  ---------------  ---------------  --------
  1  PaintRect    h              14.814814814815  14.814814814816  1e-12
  3  PaintRect    y              270.370370370370  270.370370370371  1e-12
  5  PaintText    x              12.000000000000  11.851851851852  0.148148
```

### Edge Cases

- **Op count mismatch**: If C++ emits more or fewer ops, report the first missing/extra op and its index. This likely indicates a structural divergence (e.g., Rust skips a paint call or adds an extra one).
- **State ops**: `SetCanvasColor` changes affect all subsequent ops. Flag these prominently.
- **Image pointer fields**: Skip comparison (not meaningful cross-language). Compare `img_w`/`img_h`/`img_channels` instead.

## Phase Plan

### Phase 1: Rust DrawOp Serialization

Add JSONL serialization for DrawOp. Test with `cosmos_item_border` (simplest high-diff case).

Deliverable: `DUMP_DRAW_OPS=1 cargo test --test golden cosmos_item_border` produces `target/golden-divergence/cosmos_item_border.rust_ops.jsonl`.

### Phase 2: C++ Paint Call Logging

Add conditional logging to C++ `emPainter.cpp`. Modify golden generator to enable logging.

Deliverable: `make -C tests/golden/gen run` with logging enabled produces `target/golden-divergence/cosmos_item_border.cpp_ops.jsonl`.

### Phase 3: Diff Script + First Diagnosis

Write diff script. Run on cosmos_item_border. Report exact parameter divergences.

Deliverable: Human-readable report showing exactly which parameters differ in which paint calls for cosmos_item_border.

### Phase 4: Apply to High-Diff Tests

Run the pipeline on all 8 high-diff tests. Categorize the divergence patterns.

Deliverable: Summary of divergence patterns across all high-diff tests, identifying whether it's one root cause or multiple.

## Success Criteria

1. For cosmos_item_border, the tool produces a concrete list of divergent parameters (e.g., "PaintRect at seq 3 has y=270.37 in C++ vs y=270.38 in Rust")
2. The divergences explain the observed pixel differences (replaying C++ ops through Rust painter should match C++ golden)
3. No hypotheses are formed from code reading — all findings come from the mechanical diff

## Non-Goals

- Fixing any divergence (this spec is diagnostic only)
- Modifying any widget code
- Modifying any paint primitive code
- Supporting SIMD-enabled builds (diagnostics run with SIMD off, matching FFI harness)
